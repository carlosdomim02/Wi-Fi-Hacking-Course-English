# Wireless Pentesting Lab

## Introducción
En esta sección se explicará brevemente el funcionamiento y peculiaridades del laboratorio creado con objeto de poner a prueba lo aprendido durante el siguiente curso sobre hacking de redes inalámbricas. Para una explicación más detallada de la configuración de este laboratorio se recomienda visitar la rama [`config`](https://github.com/Carlosdm06/TFG-Pentesting/tree/config), donde se hace un mayor hincapié en los distintos detalles del laboratorio.

### Esquema ilustrativo del laboratorio
![Esquema-Redes](https://github.com/user-attachments/assets/e9139ad7-6f1d-48f8-9b1e-d19ded309008)

Como se aprecia en este esquema, el laboratorio se compone principalmente de 3 subredes que pueden reflejar la estructura de red de una pequeña empresa:
- **Extranet:** representa cualquier dispositivo o servidor externo.
- **DMZ:** servidores de la empresa.
- **Intranet (y WLAN):** red interna de la empresa (sus dispositivos).

A continuación se detallará la composición y detalles de interés de cada una de estas redes.

### Extranet
Se trata de la subred más sencilla de todas, compuesta por un único dispositivo que representa cualquier computador externo a la red empresarial. Además, en esta red se puede incluir una de las interfaces del firewall que se encargará de filtrar los paquetes que entran y salen de las subredes propiedad de la empresa simulada. De esta forma, la subred queda compuesta por las siguientes direcciones IP:
- `10.5.0.1`: dirección de la máquina que representa el firewall (dispositivo que actúa de gateway para el dispositivo con IP `10.5.0.20`).
- `10.5.0.20`: dirección de la máquina que representa cualquier dispositivo externo.

El tráfico que habrá en esta red será todo aquel que provenga del firewall empresarial (o de cualquiera de sus conexiones) hacia lo que se representa como "Internet", la máquina con IP `10.5.0.20` (se expone un servidor web HTTP/HTTPS en esta para hacer pruebas de esta conexión). Además, dicha máquina podrá acceder a los servidores (o intentarlo con la red interna) de la empresa para simular conexiones desde máquinas externas a los servicios que estas ofrecen. Por otro lado, podrá intentar acceder a los dispositivos de la red interna, fallando en la mayoría de los casos, ya que el firewall se encargará de no exponer esta red al exterior (así como de restringir el acceso a los servidores). Sin embargo, esta máquina tendrá la posibilidad de formar parte de la red interna gracias a una [VPN](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#vpn) que permite la conexión de clientes autorizados (se trata de representar la conexión de empleados a través de una comunicación segura) hacia una de las máquinas de la intranet.

### DMZ
Esta red está compuesta por dos servidores web HTTP/HTTPS que hacen referencia a los posibles servidores que podrían ser propiedad de una empresa. Además, también entra en juego otra interfaz del firewall mencionado anteriormente para poder filtrar tanto las comunicaciones que entran (ya sea desde la intranet o desde la extranet) como las que salen (estas ocurrirán rara vez y se verán poco limitadas). De esta forma, el reparto de direcciones IP entre las interfaces de las diferentes máquinas mencionadas es el siguiente:
- `10.5.1.1`: dirección de la máquina que actúa en calidad de firewall.
- `10.5.1.20`: dirección de un servidor web Apache con la tecnología [Cowrie](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#cowrie) integrada.
- `10.5.1.21`: dirección de un servidor web Apache con hardening en el protocolo SSH.

Como se aprecia, se mencionan ciertas peculiaridades en estos servidores. Empezando por [Cowrie](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#cowrie), un servicio de [Honeypot](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#honeypot), listo para ser activado como una de las contramedidas que se explicarán a lo largo del curso con objeto de contrarrestar las posibles vulnerabilidades introducidas (a propósito) en el laboratorio. Por otro lado, se encuentra el hardening de las comunicaciones SSH para la administración de la segunda máquina. Esto consiste en una serie de medidas como un doble factor de autenticación (inicialmente desactivado por comodidad) o la ocultación de puertos por defecto para protocolos conocidos como SSH.

En esta subred, el tráfico está pensado principalmente para que las máquinas Apache reciban peticiones web que deban ser resueltas y devueltas, ya sea hacia las máquinas de la red interna como hacia las de la red externa. Por tanto, el resto de tráfico estará muy restringido (aunque se deja alguna vulnerabilidad, por ejemplo en contraseñas/usuarios), principalmente el tráfico SSH externo (pero no desde la red interna) que podría permitir la administración de estos servidores por usuarios ilegítimos (normalmente se desconfía de aquellos externos a la empresa, pero no de los componentes de la empresa).

### Intranet
Red interna y compuesta por dispositivos de confianza para la empresa. Esta representa la subred que más se debe proteger en el modelo DMZ, es decir, aquella zona donde las restricciones de red son más estrictas respecto a las asignadas para la subred de servidores (hay que dejar que los clientes externos puedan conectarse a estos servidores). Nuevamente, esta subred estará compuesta por varias máquinas, que representan los ordenadores cableados que podría haber dentro de la empresa, así como los dispositivos conectados de manera inalámbrica. De esta forma, cada uno de los dispositivos involucrados cuenta con una dirección IP dentro de esta subred:
- `10.5.2.1`: dirección de la interfaz del firewall que forma parte de la red interna.
- `10.5.2.20`: dirección de una máquina Ubuntu común.
- `10.5.2.21`: dirección de una máquina Ubuntu común.
- `10.5.2.22`: dirección de una máquina Ubuntu que permite las conexiones a través de VPN.
- `10.5.2.23`: dirección de una máquina con la vulnerabilidad [ShellShock](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#shellshock).
- `10.5.2.24`: dirección de la máquina que actúa como punto de acceso inalámbrico para la subred `10.5.2.128/26` (WLAN).
- `10.5.2.128/26`: subred de dispositivos inalámbricos que forman parte de la red interna.

Se puede ver que algunas de estas máquinas cuentan con ciertas tecnologías o peculiaridades, entre las que destaca la máquina que permite las conexiones hacia la red interna a través de una [VPN](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#vpn) (`10.5.2.22`). Principalmente, esta (en combinación con la tecnología VPN en la red externa) hará de servidor y cliente VPN respectivamente para simular una conexión segura en forma de protección que se detallará más adelante durante el curso. Por otro lado, se encuentra la máquina con la vulnerabilidad [ShellShock](https://github.com/Carlosdm06/TFG-Pentesting/tree/config#shellshock), la cual será explotada como un ejemplo de posible ataque. Por último, destaca la máquina que actúa como punto de acceso y que también se encuentra como parte de esta subred (conectada a la interfaz del firewall que actúa como gateway). Dicho dispositivo se encargará de unir la subred inalámbrica `10.5.2.128/26` como parte de la red interna.

La idea de esta subred es crear un ejemplo basado en la típica red interna empresarial que cuenta tanto con dispositivos cableados como con los inalámbricos. Al ser la subred más protegida, se permitirá cualquier conexión hacia fuera (y su respectiva respuesta), pero se restringirán en gran medida las conexiones entrantes en esta subred (principalmente las que permitan un escaneo de red). Por otro lado, en un principio cualquier dispositivo de esta red conformará el limitado grupo de máquinas que puedan administrar los servidores de la DMZ (luego se restringirá más como parte de las contramedidas), de forma que se permite la comunicación por SSH con las máquinas `dmz1` y `dmz2`. Sin embargo, para una mayor protección tampoco se aceptan las comunicaciones entrantes que provienen de los servidores de la empresa.

### WLAN
Esta subred consta de los dispositivos inalámbricos que forman parte de la red interna. Además, se incluye dentro de esta tanto el propio punto de acceso como una máquina atacante que inicialmente no está conectada a la red (no conoce las credenciales de acceso), pero que está en el rango adecuado para obtener señal del punto de acceso. De esta forma, se aprecia la siguiente configuración de IP según la máquina cuya interfaz inalámbrica las porta (`Ojo:` estas vienen dadas por el protocolo DHCP integrado en el punto de acceso; lo más normal es que no varíen por el orden de conexión seguido, pero puede conllevar pequeñas alteraciones):
- `10.5.2.129`: dirección de la interfaz inalámbrica del punto de acceso (este también cuenta con la dirección cableada `10.5.2.24`).
- `10.5.2.130`: dirección de una máquina Ubuntu común.
- `10.5.2.131`: dirección de una máquina Ubuntu común.
- `10.5.2.132`: dirección de una máquina Metasploitable3 (Ubuntu con vulnerabilidades).
- `10.5.2.133`: dirección de una máquina Kali Linux que representa al atacante (inicialmente no conectada pero con la posibilidad de conectarse legítimamente con esta IP para pruebas).

Se aprecia que hay dos máquinas que representan a empleados normales, así como otra que representa un dispositivo vulnerable conectado de forma inalámbrica a la red interna. Esta máquina vulnerable es una máquina Ubuntu 14.04 Metasploitable3, la cual levanta una serie de servicios vulnerables que se podrán explotar desde la máquina atacante si esta consigue acceso. Por otro lado, se encuentra la máquina Kali Linux que representa al atacante —la máquina que principalmente se controlará durante el curso para poder probar los diferentes ataques propuestos.

El tráfico en esta subred está principalmente pensado para actuar en conjunto con el resto de máquinas de la red interna, como parte de esta. Por tanto, sus objetivos serán similares, pero representando otra forma de conectarse a dicha red que cada día se está haciendo más popular, no solo en los hogares, sino en las empresas —lo que representa un mayor riesgo. Así, el tráfico estará igualmente restringido desde fuera de la intranet hacia dentro de esta, pero no al revés. Además, el tráfico entre dispositivos inalámbricos y cableados no tendrá ningún tipo de restricción, ya que se encuentran en una misma red: la intranet.

### Firewall (fw)
Este dispositivo juega un papel muy importante dentro de la red empresarial definida, suponiendo el límite entre las distintas subredes definidas (intranet, extranet y DMZ), y siendo el encargado de controlar el tráfico que circula entre cualquiera de ellas hacia otra, permitiéndolo o denegándolo. Este se encarga de definir lo que se conoce como DMZ o zona desmilitarizada; es decir, se encarga de definir distintas restricciones desde la red externa (lo que representa Internet) hacia las distintas redes propiedad de la empresa (intranet y DMZ o red de servidores). De esta forma, limitará más el tráfico entrante (pero no el saliente) hacia la red interna, desde cualquiera de las otras redes. En cambio, la limitación desde cualquier red hacia la DMZ o red de servidores se ve relajada, ya que estos deben dar servicio tanto a los dispositivos de los trabajadores como a aquellos que son ajenos a la empresa, aunque siempre con unas restricciones mínimas que eviten ataques. En especial, se relajarán estas restricciones para las comunicaciones provenientes de la red interna, ya que esta debe ser capaz de administrar los dispositivos que actúan como servidores (esto se restringe aún más desde el propio firewall durante el curso).

Por otro lado, el firewall será el dispositivo que actúe como gateway de todos los demás (excepto de aquellos que conforman la red inalámbrica, aunque lo hará de forma indirecta al ser su gateway el punto de acceso, que a su vez tiene al firewall como gateway). Esto hace que todos los dispositivos vean al firewall como primer salto en las comunicaciones, facilitando la administración y filtrado de paquetes que este implementa. Además, cuenta con la capacidad de enmascarar las direcciones IP de la red interna, dificultando aún más la labor de los atacantes externos que intentan hacer un escaneo de redes para conocer la estructura interna de la red empresarial. Sin embargo, este enmascaramiento o NAT no se realiza para los dispositivos que representan a los servidores HTTP/HTTPS, ya que representan IPs públicas que deberían ser visibles desde fuera (aunque se podría implementar un equilibrador de carga/proxy inverso que exponga su IP y oculte la interna de estos dispositivos). La idea de que incluso la red externa tenga este firewall como gateway es representar la forma de actuación común en la que el único punto de acceso externo hacia la red empresarial sea el firewall, permitiendo así el control del tráfico entrante.

Finalmente, se aprecia que este firewall no deja de ser una máquina Ubuntu 24.04 con diferentes reglas (iptables) que permiten el control y filtrado de los paquetes que lo atraviesan (aquellos que circulan entre las distintas subredes explicadas). Por tanto, representa una máquina que puede suponer una alta sobrecarga, siendo objeto de ataques de tipo denegación de servicio (DoS). Con esto en mente, se limitan las comunicaciones a través de ICMP (`echo-request`) de tal forma que, si se detectan más de 10 paquetes de golpe, se limita la entrada de paquetes de este tipo durante un tiempo a 5 paquetes por minuto.

### Objetivo del laboratorio
Este laboratorio tiene como principal objetivo servir como entorno de pruebas de los distintos ataques que se comentarán durante el desarrollo del curso. Principalmente, dichos ataques consisten en romper los protocolos de seguridad (WEP, WPA/WPA2, WPS) del estándar Wi-Fi (`IEEE 802.11`) con la ayuda de ciertas herramientas típicas de máquinas Kali Linux. De esta forma, el laboratorio será usado para poner a prueba las distintas herramientas y ataques que se expliquen durante el curso con la idea de vulnerar la seguridad de estos protocolos y conseguir un acceso ilegítimo en la red inalámbrica interna de la empresa simulada. Aunque no solo se quedará ahí la propuesta del curso, sino que propondrá ejemplos de ataques que se pueden desarrollar una vez se consigue dicho acceso, viendo así los riesgos reales que puede causar dicho acceso ilegítimo. Con esto se busca ofrecer un laboratorio para desarrollar las habilidades de penetración (así como la localización de los riesgos y contramedidas que se pueden aplicar para paliar los efectos adversos) de personas que se inician en el mundo de la seguridad ofensiva, siempre con el objetivo de enseñar a futuros (o ya consolidados) auditores de seguridad que usen estos conocimientos para detectar fallas de seguridad antes de que lo hagan usuarios malintencionados.
