## Introduction
This section briefly explains the operation and key characteristics of the lab created to test the knowledge acquired throughout the following wireless network hacking course. For a more detailed explanation of this lab's configuration, it is recommended to visit the [`config`](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config) branch, where much greater emphasis is placed on the various details of the lab.

### Illustrative Lab Diagram
![Esquema-Redes](https://github.com/user-attachments/assets/e9139ad7-6f1d-48f8-9b1e-d19ded309008)

As shown in this diagram, the lab is mainly composed of 3 subnets that can reflect the network structure of a small company:
- **Extranet:** represents any external device or server.
- **DMZ:** the company's servers.
- **Intranet (and WLAN):** the company's internal network (its devices).

The composition and relevant details of each of these networks are described below.

### Extranet
This is the simplest subnet of all, composed of a single device that represents any computer external to the corporate network. In addition, one of the firewall's interfaces can be included in this network, and it will be responsible for filtering the packets entering and leaving the subnets belonging to the simulated company. In this way, the subnet is made up of the following IP addresses:
- `10.5.0.1`: address of the machine representing the firewall (the device acting as the gateway for the device with IP `10.5.0.20`).
- `10.5.0.20`: address of the machine representing any external device.

The traffic in this network will be everything that comes from the corporate firewall (or any of its connections) toward what is represented as "the Internet," the machine with IP `10.5.0.20` (an HTTP/HTTPS web server is exposed on it for testing this connection). In addition, this machine will be able to access the company's servers (or attempt to access the internal network) in order to simulate connections from external machines to the services they provide. On the other hand, it may attempt to access devices on the internal network, failing in most cases, since the firewall will ensure that this network is not exposed to the outside world (as well as restricting access to the servers). However, this machine will have the possibility of becoming part of the internal network thanks to a [VPN](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#vpn), which allows authorized clients to connect (this is intended to represent employees connecting through secure communication) to one of the machines on the intranet.

### DMZ
This network is composed of two HTTP/HTTPS web servers that represent possible servers owned by a company. In addition, another interface of the previously mentioned firewall also comes into play in order to filter both incoming communications (whether from the intranet or the extranet) and outgoing communications (these will occur rarely and will face few restrictions). Thus, the distribution of IP addresses among the interfaces of the different machines mentioned is as follows:
- `10.5.1.1`: address of the machine acting as the firewall.
- `10.5.1.20`: address of an Apache web server with [Cowrie](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#cowrie) technology integrated.
- `10.5.1.21`: address of an Apache web server with SSH protocol hardening.

As can be seen, certain peculiarities are mentioned for these servers. Starting with [Cowrie](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#cowrie), a [honeypot](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#honeypot) service that is ready to be activated as one of the countermeasures that will be explained throughout the course in order to counter the possible vulnerabilities deliberately introduced into the lab. On the other hand, there is the hardening of SSH communications for administering the second machine. This consists of a series of measures such as two-factor authentication (initially disabled for convenience) or hiding default ports for well-known protocols such as SSH.

In this subnet, traffic is mainly intended for the Apache machines to receive web requests that must be processed and returned, whether to machines on the internal network or to those on the external network. Therefore, the rest of the traffic will be heavily restricted (although some vulnerabilities are left in place, for example in passwords/users), mainly external SSH traffic (but not from the internal network), which could allow these servers to be administered by illegitimate users (components external to the company are normally distrusted, while components belonging to the company are not).

### Intranet
This is the internal network and is composed of devices trusted by the company. It represents the subnet that must be protected the most in the DMZ model; that is, the area where network restrictions are stricter than those assigned to the servers' subnet (external clients need to be allowed to connect to those servers). Once again, this subnet is made up of several machines, representing the wired computers that could exist within the company as well as the devices connected wirelessly. Thus, each of the devices involved has an IP address within this subnet:
- `10.5.2.1`: address of the firewall interface that is part of the internal network.
- `10.5.2.20`: address of a standard Ubuntu machine.
- `10.5.2.21`: address of a standard Ubuntu machine.
- `10.5.2.22`: address of an Ubuntu machine that allows VPN connections.
- `10.5.2.23`: address of a machine with the [ShellShock](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#shellshock) vulnerability.
- `10.5.2.24`: address of the machine acting as the wireless access point for the `10.5.2.128/26` subnet (WLAN).
- `10.5.2.128/26`: subnet of wireless devices that are part of the internal network.

Some of these machines feature certain technologies or characteristics. Particularly notable is the machine that allows connections to the internal network through a [VPN](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#vpn) (`10.5.2.22`). Primarily, this machine (in combination with the VPN technology in the external network) will act as VPN server and VPN client respectively in order to simulate a secure connection as a form of protection that will be described later during the course. On the other hand, there is the machine with the [ShellShock](https://github.com/carlosdomim02/Wi-Fi-Hacking-Course-English/tree/config#shellshock) vulnerability, which will be exploited as an example of a possible attack. Finally, the machine acting as the access point stands out, since it is also part of this subnet (connected to the firewall interface acting as gateway). This device will be responsible for integrating the wireless subnet `10.5.2.128/26` as part of the internal network.

The idea behind this subnet is to create an example based on a typical corporate internal network that includes both wired and wireless devices. Since it is the most protected subnet, any outbound connection (and its corresponding response) will be allowed, but incoming connections to this subnet will be heavily restricted (mainly those that enable network scanning). On the other hand, initially any device on this network will form the limited group of machines allowed to administer the DMZ servers (this will later be restricted further as part of the countermeasures), so SSH communication with machines `dmz1` and `dmz2` is allowed. However, for greater protection, inbound communications originating from the company's servers are also not accepted.

### WLAN
This subnet consists of the wireless devices that are part of the internal network. In addition, it includes both the access point itself and an attacker machine that is initially not connected to the network (it does not know the access credentials), but is within proper range to receive the access point's signal. Thus, the following IP configuration can be seen depending on the machine whose wireless interface carries them (`Note:` these are provided by the DHCP protocol integrated into the access point; normally they do not vary depending on connection order, but slight changes may occur):
- `10.5.2.129`: address of the access point's wireless interface (it also has the wired address `10.5.2.24`).
- `10.5.2.130`: address of a standard Ubuntu machine.
- `10.5.2.131`: address of a standard Ubuntu machine.
- `10.5.2.132`: address of a Metasploitable3 machine (Ubuntu with vulnerabilities).
- `10.5.2.133`: address of a Kali Linux machine representing the attacker (initially disconnected, but with the possibility of connecting legitimately using this IP for testing).

It can be seen that there are two machines representing normal employees, as well as another representing a vulnerable device wirelessly connected to the internal network. This vulnerable machine is an Ubuntu 14.04 Metasploitable3 machine, which exposes a series of vulnerable services that can be exploited from the attacker machine if it gains access. On the other hand, there is the Kali Linux machine representing the attacker—the machine that will mainly be controlled during the course in order to test the various proposed attacks.

Traffic in this subnet is mainly intended to act together with the rest of the machines on the internal network, as part of it. Therefore, its objectives will be similar, but representing another way of connecting to that network, which is becoming increasingly popular every day—not only in homes, but also in companies—which represents a greater risk. Thus, traffic from outside the intranet to inside it will also be restricted, but not the other way around. In addition, traffic between wireless and wired devices will have no restriction whatsoever, since they are on the same network: the intranet.

### Firewall (fw)
This device plays a very important role within the defined corporate network, representing the boundary between the different defined subnets (intranet, extranet, and DMZ), and being responsible for controlling the traffic flowing between any of them toward another, allowing or denying it. It is in charge of defining what is known as a DMZ, or demilitarized zone; that is, it defines different restrictions from the external network (what represents the Internet) toward the different networks owned by the company (intranet and DMZ or server network). Thus, it will place greater restrictions on incoming traffic (but not outgoing traffic) toward the internal network from any of the other networks. In contrast, restrictions from any network toward the DMZ or server network are relaxed, since these servers must provide services both to employees' devices and to devices external to the company, though always with minimum restrictions to prevent attacks. In particular, these restrictions are relaxed for communications originating from the internal network, since it must be able to administer the devices acting as servers (this is restricted even further from the firewall itself during the course).

On the other hand, the firewall will act as the gateway for all other devices (except for those that make up the wireless network, although it will do so indirectly since their gateway is the access point, which in turn has the firewall as its gateway). This makes all devices see the firewall as the first hop in communications, facilitating administration and packet filtering implemented by it. In addition, it has the ability to mask the IP addresses of the internal network, making the job of external attackers who attempt to scan networks to discover the internal structure of the corporate network even harder. However, this masking, or NAT, is not performed for the devices representing the HTTP/HTTPS servers, since they represent public IPs that should be visible from the outside (although a load balancer/reverse proxy could be implemented to expose its own IP and hide the internal one of these devices). The idea that even the external network has this firewall as its gateway is to represent the common behavior in which the firewall is the only external access point into the corporate network, thus allowing control of incoming traffic.

Finally, it can be seen that this firewall is still just an Ubuntu 24.04 machine with different rules (`iptables`) that enable control and filtering of the packets that pass through it (those that circulate between the different subnets explained above). Therefore, it represents a machine that may be subject to a high workload, making it a target for denial-of-service (DoS) attacks. With this in mind, communications through ICMP (`echo-request`) are limited in such a way that, if more than 10 packets are detected at once, the entry of packets of this type is limited for a period to 5 packets per minute.

### Objective of the Lab
The main objective of this lab is to serve as a testing environment for the various attacks that will be discussed throughout the course. These attacks mainly consist of breaking the security protocols (WEP, WPA/WPA2, WPS) of the Wi-Fi standard (`IEEE 802.11`) with the help of certain tools commonly found on Kali Linux machines. In this way, the lab will be used to test the different tools and attacks explained throughout the course, with the goal of compromising the security of these protocols and gaining illegitimate access to the company's simulated internal wireless network. However, the course proposal does not stop there; it also proposes examples of attacks that can be developed once such access is obtained, thereby showing the real risks that this illegitimate access can cause. The aim is thus to provide a lab to develop the penetration skills of people who are entering the world of offensive security (as well as the ability to identify risks and countermeasures that can be applied to mitigate adverse effects), always with the objective of teaching future (or already established) security auditors to use this knowledge to detect security flaws before malicious users do.
